The "Debug & Rescue" Mission

Diagnose, fix, and resume crashed training runs

This demo illustrates how GREMLIN transforms debugging from a forensic exercise into a guided recovery process.

$5,000 Saved
Per avoided restart from Epoch 0

1. The Crash Report: "Forensic Analysis"

Understanding why a 128-GPU job crashed overnight.

Input:

"Why did job 'job-fusion-xl-v2' fail?"

Insight: No More Log Archaeology

Instead of scrolling through 10,000 lines of logs, GREMLIN:

  • Parses Stack Traces: Identifies the exact error type (OOM)
  • Correlates Events: Links to node heartbeats from Hypernexus
  • Diagnoses Root Cause: Pinpoints the specific operation (contrastive_loss) and location (Rank 7)
  • Rules Out Hardware: Confirms all nodes are healthy

2. The Fix & Resume: "The Save"

Recovering without losing progress.

Input:

"Fix the OOM and resume training."

Key Concepts
  • Smart Fix Suggestion: GREMLIN knows to reduce val_batch_size, not training batch size.
  • Checkpoint Discovery: Finds the latest valid checkpoint in Atlas (step 4,500).
  • Complete Action Plan: Clone, patch, and resubmit in one flow.

3. Execute Recovery

One-click resume from checkpoint.

Input:

"Y" (Execute fix)

$5K in Compute Saved

Without GREMLIN:

  • Manually edit YAML config
  • Hunt for GCS checkpoint path
  • Risk typos in resume_from field
  • Possibly restart from Epoch 0

With GREMLIN: Resume from step 4,500 with validated config changes.

4. Proactive Monitoring: "The Watch"

Setting up alerts to prevent future surprises.

Input:

"Monitor this run and alert me if anything looks wrong."

Key Concepts
  • Smart Thresholds: 2x moving average for loss spikes, not absolute values.
  • Tiered Alerts: Warnings vs. critical alerts depending on severity.
  • Slack Integration: Notifications go directly to the scientist.

5. Health Check

Verifying the resumed job is healthy.

Input:

"How is the resumed job doing?"

Key Concepts
  • Expected vs. Actual: Compare metrics against historical baselines.
  • At-a-Glance Status: Green/yellow/red indicators for quick assessment.
  • Completion Estimate: Know exactly when the job will finish.

6. Knowledge Base: Common Errors

Learning Tempus-specific failure patterns.

Input:

"What common errors should I watch out for on Hypernexus?"

Insight: Tempus-Specific, Not StackOverflow

Generic error messages get Tempus-specific solutions:

  • NCCL timeout: Knows Hypernexus uses TCP/IP (no RDMA)
  • Ray actor died: Knows about Spot instance patterns
  • gcsfuse errors: Knows internal caching behavior
