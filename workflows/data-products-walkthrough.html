Data Products Integration

From ad-hoc CSV files to immutable, versioned datasets

This demo illustrates how GREMLIN bridges the gap between raw data and reproducible ML experiments through the Tempus Data Products platform.

Reproducibility First
Immutable datasets with full lineage tracking

1. Discovery: "What's in the Pantry?"

Finding available Data Products for your research.

Input:

"Find Data Products for Non-Small Cell Lung Cancer with molecular data."

Key Concepts
  • Natural Language Search: No need to know exact Data Product names or navigate complex UIs.
  • Classification Visibility: See data sensitivity levels (SECRET/PHI) upfront.
  • Version Awareness: Know exactly which version you're getting.

2. Schema Exploration

Understanding data structure before writing queries.

Input:

"describe clinical_nsclc_outcomes"

Insight: Self-Service Schema Browser

GREMLIN provides instant access to:

  • Full Schema: Field names, types, and descriptions.
  • Required vs Optional: Know what data is guaranteed vs. potentially null.
  • Your Access Level: Verify permissions before writing queries.

3. Access & Join Generation

Requesting access and generating validated queries.

Input:

"I need to join clinical_nsclc_outcomes with rna_expression_nsclc for patients with 1L data."

Automated Access Workflow
  • Permission Check: GREMLIN verifies access before you start working.
  • Auto-Request: Missing permissions trigger automatic access requests.
  • Query Pre-Generation: Get the validated SQL while waiting for approval.

4. Pin & Version: "Seal the Jar"

Creating immutable dataset snapshots.

Input:

"Pin this cohort as 'nsclc_fusion_train_v1' for my experiment."

Key Concepts
  • Immutable Artifacts: SHA-pinned datasets that never change.
  • Full Lineage: Track exactly which source Data Products were used.
  • Atlas Registration: Proper URI (atlas://datasets/...) for experiment configs.

5. Audit: "Where Did This Come From?"

Tracing model provenance for debugging.

Input:

"What data was model 'nsclc_fusion_prod_v2' trained on?"

Insight: Proactive Issue Detection

  • Deprecation Alerts: Flags when source data has been updated.
  • Bug References: Links to JIRAs explaining what changed.
  • Impact Assessment: Suggests when retraining is warranted.

6. Differential Analysis

Comparing data versions before retraining.

Input:

"Y" (Compare versions)

Key Concepts
  • Quantitative Diff: Patient counts, gene coverage, distribution changes.
  • Actionable Recommendations: Clear guidance on whether to retrain.
  • One-Click Updates: Create new cohorts with updated versions instantly.
